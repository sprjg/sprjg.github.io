LaplaceIO
Intuitive whiteboarding

Summary

  • As part of documentating architectures at Tonbo, I used a lot of flowchart software
  • Draw.io was extremely easy to use and extremely poor to scale
  • Graphviz was super scalable, but not user friendly
  • It made sense that Graphviz could be a backend to a very friendly and intuitive UX
  • I wanted a UX that would understand user strokes and perform the appropriate action. A roughly drawn shape = node with that shape, a tap inside a shape = text input for that node, a line between two blocks = a connection between the two, and so on
  • Explored SVG, Canvas as frontend, React was the natural choice to code things up
  • Ultimately the scope was too ambitious. I learned deploy heavy applications in a production environment but the stated objective was never reached

Technologies used:

  • React
  • Django
  • Socketio

Seen
CLI lookup for hypothes.is

Summary

  • I spend most of my time on the terminal
  • There are certain pages/guides that I frequently used as reference. Could be stackoverflow answers that were useful, could be keybindings for vim
  • I hoped to remove the context switch from terminal to a browser and to look up the bookmarks
  • Surfraw already helps majorly in this. You can quickly search in google from your terminal if you are comfortable with elinks, w3m
  • I wanted something more: could I get access to snippets I'd highlighted from the terminal itself?
  • Hypothes.is already has a wonderful service that allows you to highlight webpages. And the best part is the API
  • This is a Go implementation of their API. It shows the highlights and I can quickly search through them. The highlighted parts are readily available from the terminal but I can also open the links for further study

Technologies used:

  • Go

Wysgy
Command line note-taking tool

Summary

  • This was a second attempt at creating a note taking tool, this time on the terminal
  • My objective was a tool that stores notes and also the relations between notes
  • This is in some way possible with just a folder structure. I was hoping for a more graph oriented approach where notes can be related to multiple other notes. The relations themselves being notes
  • I chose Rust because this was a CLI that would interact significantly with the underlying file system. I'd heard about the promises that Rust compiler makes, particularly in race conditions and memory management
  • Ultimately the GUI was not as comfortable as I was hoping for
  • I also discovered Roam, then subsequently org-mode, then org-roam, all of which solved note taking with far greater scope
  • Ultimately this project was a nice entry point into Rust, which I'll continue to use for systems programming

Technologies used:
